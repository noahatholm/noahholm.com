Project Proposal

In second year at the University of Nottingham, all computer science students are required to participate in a group project that spans the entire year and is worth 20 credits overall. The projects are decided by a series of industry partners and internal stakeholders.

In September, we were assigned our randomly allocated group, which consisted of eight students in total. This was given to us in the form of an email containing the names of our teammates. I subsequently reached out and arranged a "pub trip" for us all to meet up, nothing technical.

Following our pub trip, we were tasked with choosing roles amongst ourselves. The three roles were Team Leader, Team Admin, and Git Admin. I knew that I wanted to go for a role but was split between Leader and Admin. However, after reading the project handbook, I came to the conclusion that Leader had no official responsibilities and was just a formality, whereas the Admin was responsible for submitting all reports, bidding for projects, communicating with stakeholders, and more. From this, I made the decision that Team Admin was the most important role overall, so I applied for it. Fortunately, I was voted in by my peers.

The next step was to bid for our project. Prior to bidding, all project briefs are made available online for us to read and decide which ones we prefer. Next, the Team Admin has to pick three projects on a first come, first served basis, using a Moodle quiz that opens at 9AM (Fortunately, my concert ticket buying experience came in clutch here). This was stressful, but I managed to get two of our first choices, leaving us with Memories, UniDays GO, and Quantum Tetris.

Now that we had chosen our projects, we still needed to create a pitch for each one, which included a short video and a document explaining why we wanted the project and our execution plan. This was then presented to the project sponsors, who would choose which group they'd like to carry out the project. As you can tell, we were chosen for Memories.

Project Planning

Now that we had our project secured, we had to plan out the functional and non-functional requirements, as well as our technical stack:

  • Backend: Python (the go-to language for ML and AI)
    • Python 3.11 (chosen because of NumPy V2 compatibility)
    • FastAPI (performance and lightweight)
    • Multiple ML and non-ML methods used for data extraction:
      • Image Classification CNN
      • Emotion Detection ViT
      • Facial Recognition MTCNN
      • Facial Embedding CNN
      • Optical Character Recognition CRNN
      • Metadata Extraction
  • Frontend: React.js (widely used)
  • Database: Supabase (free, hosted, and has vector support)
  • Deployment:
    • Reverse Proxy: NGINX
    • Virtualization: Docker
    • Image Storage: AWS S3
    • Hosting: AWS EC2
    • CDN: AWS CloudFront

Development

I chose a Scrumban development methodology that combines the iterative sprints from Scrum with the continuous workflow of Kanban. This worked really well using 2 week sprints, with each one having its own goals and subgoals.

I primarily worked on the backend logic and deployment, which admittedly included a lot of trivial API calls, so I'm only going to talk about the bespoke (and more interesting) things I worked on.

Tagging Algorithm

Having implemented all the AI methods listed above, I needed a way to call them all on a single image and collect the returned data. Here, I used a heavily object-oriented approach to encapsulate all the separate methods into a single class called Tagger.

To run this safely, I used a producer consumer design pattern to ensure that only one tagging job happens at a time. Because it is highly resource intensive, a lot of concurrent calls would cause us to run out of memory and CPU, freezing the website and making it unresponsive. The one problem is that despite running it in a separate user thread using asyncio, it is still locked to a single system thread due to Python's GIL. To reimplement this in the future, I would make it a separate process that runs on its own core and communicates using a message queue.

Categorising Algorithm

All extracted data now needs to be processed to cluster images into "memory boxes". This clustering is based on shared individuals, locations, dates, emotions, and other embedded metadata.

I began by writing out four key requirements that the algorithm needed to achieve:

  1. The algorithm should be run on one image at a time immediately after upload.
  2. It must place the image into existing boxes and/or create new boxes for the image and related images if a matching box doesn’t exist yet.
  3. The types of boxes it creates are location, emotion, face, season, and date.
  4. A new box shouldn't be created unless it contains at least three images.

Deployment

The project was containerised into three separate containers: Backend, Frontend, and App ( handles SSL and NGINX).

Subsequently, this architecture made it incredibly easy to deploy on an AWS EC2 instance. I made use of the AWS Free Tier and promotional credits, which gave us more than enough overhead to host the site for the entire duration of the academic project without incurring costs.

I also purchased the official domain www.yourmemories.app for the live launch, though I'm still deciding on how long I'll it hosted.

Evaluation

Overall, this project was a fantastic experience. It dramatically leveled up my skills as a developer, and I learned an immense amount particularly when it came to configuring cloud infrastructure. Being able to talk about the system architecture extensively in technical interviews has been invaluable, and I honestly doubt I would have landed my upcoming software engineering opportunities without it.
